## 为什么要有自己的网站？

社交媒体来来去去，但个人网站是真正属于自己的领地。它不受平台规则限制，不会被算法埋没，也不会因为某个平台倒闭而消失。

对开发者来说，个人网站还是最好的作品集——它本身就是你能力的证明。

## 技术选型的纠结

一开始我在几个方案之间犹豫：

- **Next.js** — 功能最全，但对博客来说太重了
- **纯 HTML/CSS** — 最轻量，但维护成本高
- **Astro** — 专为内容站点设计，静态优先，按需加载 JS

最终选了 Astro，因为它的 Islands 架构完美解决了"大部分页面是静态的，但某些区域需要交互"的场景。

## 双语支持比想象中复杂

我希望网站同时支持中文和英文。听起来简单，做起来有不少细节：

- URL 结构要清晰：`/zh/blog/xxx` 和 `/en/blog/xxx`
- 每篇文章需要一个 `translationSlug` 字段来关联两种语言版本
- UI 文字需要翻译文件，不能硬编码
- 根路径需要检测浏览器语言自动跳转

最终用文件路由 + JSON 翻译文件的方案，简单直接。

## 主题切换的小陷阱

双主题（日间/夜间）看起来只是换几个颜色，但有一个经典问题：页面加载时的闪烁。

如果主题 JS 在页面渲染后才执行，用户会先看到默认主题再跳到保存的主题。解决方法是在 `<head>` 里放一段内联脚本，在 DOM 渲染前就设好 `data-theme` 属性。

## 部署到 GitHub Pages

这是最省心的部分。写一个 GitHub Actions workflow，push 到 main 分支就自动构建部署，30 秒上线。免费，没有服务器维护。

唯一要注意的是把 GitHub Pages 的 source 设置为 "GitHub Actions" 而不是 "Deploy from a branch"。

## 关于 Pretext 的实验

这次最有趣的部分是用 Pretext 做文字交互效果。它让我可以在 Canvas 上精确控制文字位置，实现了传统 CSS 做不到的排版——比如文字环绕一个像素小人。

不过也踩了一些坑：Pretext 需要浏览器环境，在 SSR 时不可用，需要用 try/catch 包裹并提供 fallback。

## 总结

搭建个人网站不需要追求完美，先上线再迭代。一个有个性的简单页面，比一个永远在"规划中"的完美方案有价值得多。
