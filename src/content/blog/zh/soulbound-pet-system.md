## 灵感来源

Claude Code 有一个彩蛋叫 Buddy——终端里的 ASCII 艺术小生物。每次打开终端，你的 buddy 就在那里，用 Unicode 字符拼出一只猫、一条龙或一只水母，在扫描线和荧光辉光中缓缓浮动。

看到这个我就想：我的 AI 角色陪伴 app 也应该有类似的东西。不只是装饰，而是一个完整的收集养成系统——每个角色卡绑定一只宠物，聊天就能让它变强。

## 确定性哈希：同一张卡永远是同一只宠物

核心问题是：怎么给角色卡分配宠物？

随机？不行。用户把卡分享给朋友，朋友导入后应该看到同一只宠物（只是 XP 从零开始）。所以需要确定性——同一个 card ID 永远生成同一个结果。

方案是 FNV-1a 哈希 + Mulberry32 伪随机数生成器：

```
cardId (UUID)
  → FNV-1a("soulbound-pet-" + id)
  → Mulberry32 PRNG seed
  → roll 1: 是否闪光 (2%)
  → roll 2: 稀有度 (Common 50% / Rare 28% / Epic 15% / Legendary 5%)
  → roll 3: 种类 (从稀有度池中均匀选取)
  → roll 4-8: 初始属性值 (±15% 方差)
```

这两个算法加起来不到 30 行 Swift，零依赖。同一个 UUID 跑一百万次，结果完全一样。

## 64 只宠物的 ASCII 帧

每只宠物有 3 帧动画，用 Unicode box-drawing 字符 + 表情符号拼出来。比如泡泡灵：

```
帧 1:        帧 2:        帧 3:
 ╭───╮        ╭───╮        ╭───╮
( ･ ･ )      ( ─ ─ )      ( ･ ･ )
 ╰─∪─╯        ╰─∪─╯        ╰─◡─╯
  ≋≋≋          ≋≋≋         ≋≋≋≋≋
 ○  ○          ○  ○       ○     ○
```

64 只宠物分 4 个稀有度：28 Common、20 Rare、12 Epic、4 Legendary。从泡泡灵到星鲸灵，从错兔灵到霓武灵。

渲染用的是打包进 app 的 JetBrains Mono 字体（OFL 免费商用），Info.plist 注册 `UIAppFonts`。iOS 对缺失字形会自动 fallback 到系统字体，不会出方块。

三层 `.shadow()` 叠加模拟终端磷光屏辉光，闪光宠物再加一层粉色光晕。加上一个 `easeInOut` 的浮动动画，ASCII 字符就有了呼吸感。

## 养成系统：聊天 → 属性成长

5 个属性：亲密、活力、灵感、治愈、幸运。

每次聊天结束（app 进入后台），系统会分析对话关键词：

- 用户说"谢谢""爱""开心" → 亲密 +1~2
- 用户分享私密话题"秘密""信任" → 幸运 +1~2
- 对话内容丰富多样 → 灵感 +1~2
- 聊得久（消息多） → 活力 + 治愈

这个估算完全是规则驱动的，不调 LLM，零成本。粗糙但够用——重点是让"和角色聊天"这件事有反馈循环。

属性上限由宠物种类的 `baseMax` 决定，升星可以提高上限（★1 +10%，★2 +22%，★3 +35%）。

## 进化：代码生成装饰

4 阶进化：幼体 → 成长 → 成熟 → 觉醒。

64 只宠物 × 4 阶段 × 3 帧 = 768 帧手工 ASCII？不可能。

解法是 `PetEvolutionDecorator`——代码自动给 ASCII 帧加装饰：

- **幼体**：原始帧，不修改
- **成长**：顶部加一个 ✦
- **成熟**：包一层单线边框 `┌✦───✦┐`
- **觉醒**：双线光环 `✦═══✦` + 眼睛字符升级（`･` → `◈`）

同一套基础帧，四种视觉效果，零额外美术成本。

## 碎片升星

宠物池只有 64 只，角色卡可以无限创建。重复的怎么办？

重复生成已拥有的种类 → 自动转为碎片。碎片攒够一定数量可以升星：

| 稀有度 | 碎片/次 | 升星成本 |
|--------|---------|---------|
| Common | 3 | 5 |
| Rare | 5 | 8 |
| Epic | 8 | 12 |
| Legendary | 15 | 20 |

最高 ★3，每颗星提升属性上限。这给了用户两个理由去创建更多角色卡：新宠物，或者给老宠物升星。

## SwiftData 持久化

宠物身份（哪只宠物）是由 hash 确定的，不需要存储——每次从 card ID 重新算就行。

宠物养成数据（XP、属性、进化阶段、星级、碎片）存在 SwiftData 的 `@Model PetData` 里。查询用 speciesId 而不是 characterId，因为所有权是全局的——你拥有的是"泡泡灵"这个种类，不是"某个角色的泡泡灵"。

一个微妙的 bug：最初用 `characterId` 查宠物，导致非首创角色找不到对应宠物。改成先 hash 出 speciesId 再查就对了。

## 集成点

宠物出现在 4 个地方：

1. **聊天界面**：左下角半透明小宠物，点击弹出属性 popover
2. **首页**："My Pets" 网格卡片，显示收集进度 "N/64 collected"
3. **收藏页**：网格浏览 + 稀有度过滤 + 碎片库入口
4. **角色导入**：导入角色卡自动生成宠物，首次聊天也会触发

新宠物有紫色 toast 提醒，重复的显示 "+N shards"。

## 写在最后

整个系统 16 个 commit，12 个新文件。最大的文件是宠物注册表（3 个文件共 800+ 行，全是 ASCII 帧数据）。

核心设计原则是**零 LLM 成本**——所有计算（hash、属性增长、关键词匹配、进化装饰）都是纯本地的。宠物系统不应该为了一个游戏化功能去消耗用户的 API 额度。

下一步可能是宠物技能影响聊天体验（亲密度高 → 角色更亲昵？），但现在先让它安安静静地浮在聊天界面左下角，等用户发现它、点它、养它。
